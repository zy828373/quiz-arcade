# 09（Codex号池）单平台版 V2 改造蓝图

## 1. 这份蓝图解决什么问题

这份文档不是讨论“理想中最先进的架构”，而是结合你现在仓库的真实状态，回答 3 个实际问题：

1. 当前为什么会觉得 V1 / V2 来回切换很麻烦
2. 现在能不能合成一个“大平台”
3. 最适合当前项目的合并路线应该怎么走

这份蓝图的核心判断只有一句话：

**现在最应该做的，不是立刻重写底层引擎，而是先让 V2 成为唯一可见的平台，让 Team Pool 退到后台作为受管执行引擎。**

---

## 2. 当前项目的真实状态

结合当前仓库代码和运行方式，系统实际上分成 4 层：

### 2.1 底层执行引擎

当前真正负责“出结果”的核心仍然是：

- `cli-proxy-api.exe`
- `config_team.yaml`
- `start_team.bat`
- 端口 `8317`

这一层当前不能直接删掉。  
V2 的 OpenAI / Anthropic 请求，目前仍然主要是：

`V2 Gateway -> Team Pool (:8317)`

### 2.2 旧入口层

这一层主要解决外部兼容和旧接入路径：

- `anthropic_proxy.js`
- `start_anthropic_proxy.bat`
- `cloudflared.exe`
- `start_tunnel.bat`
- 端口 `8320`
- Cloudflare Tunnel

如果当前目标是“本机自用优先”，这一层可以降级为可选层。

### 2.3 V2 控制层 / 新前门

这一层已经成型，是当前最有价值的部分：

- `v2/`
- `/ops`
- `/control/*`
- `/health/*`
- `/v1/chat/completions`
- `/v1/messages`
- readiness gate
- synthetic probe
- cutover / rollback

它已经不是“辅助页面”，而是新的控制平台和统一前门。

### 2.4 运维脚本层

这一层把系统串起来：

- `start_all.bat`
- `health_check.ps1`
- `enter_parallel.ps1`
- `enter_canary.ps1`
- `enter_primary.ps1`
- `rollback_legacy.ps1`
- `start_v2.bat`

当前这层的最大问题不是功能少，而是“平台能力已经进了 V2，但很多启动和管理动作还留在脚本里”。

---

## 3. 为什么你现在会觉得很别扭

因为系统当前是一个“过渡态”：

- 你看到和操作的是 V2
- 真正执行很多请求的是 Team Pool
- 旧入口层还没有完全退出
- 进程管理还主要在脚本和窗口里

所以你现在会出现这些体验问题：

- 要记多个端口：`8317`、`18320`、有时还有 `8320`
- 要理解谁是控制层，谁是执行层
- 要区分“页面通过”和“底层真的能出结果”
- 要看多个窗口而不是一个平台

这不是你的问题，是当前架构本来就还没有完全收口。

---

## 4. 能不能合成一个大平台

### 4.1 结论

**可以，而且应该。**

但这里的“合”要分成两种：

### 4.2 能立刻做的合并：操作层合并

这一步的目标是：

- 只保留一个你真正面对的平台
- 只保留一个页面
- 只保留一个启动入口
- 只保留一套巡检和状态说明

即使底层仍然是：

- `V2` 负责控制和前门
- `Team Pool` 负责执行

你也不应该继续手动来回切窗口。

**这是现在最应该做的合并。**

### 4.3 暂时不建议硬做的合并：执行层合并

这一步才是：

- 去掉 `cli-proxy-api.exe`
- 让 V2 自己接管底层执行和认证态
- 彻底摆脱 Team Pool

当前不建议直接做，原因很现实：

- 你现在已经验证出 `8317 /v1/models` 能通，但 `8317 /v1/chat/completions` 在某些时候会失败
- 说明 Team Pool 里仍然握着关键的认证执行逻辑
- 如果现在强行硬并，等于要重写执行引擎，不是“平台整合”，而是“引擎重构”

**这一步以后可以做，但不该是现在的第一目标。**

---

## 5. 最适合当前项目的总体路线

### 总原则

**先合“人看到的和人操作的”，再考虑是否合“底层执行引擎”。**

换句话说：

- 先让 V2 成为唯一平台
- 先把 Team Pool 藏到后台
- 先把旧脚本收编进 V2
- 以后再决定是否吞掉 Team Pool

---

## 6. 目标形态：单平台版 V2

### 6.1 你未来应该只面对一个东西

未来你应该只面对：

- 一个页面：`/ops`
- 一个主入口：`18320`
- 一个运行模式选择
- 一套巡检按钮
- 一套状态说明
- 一个回滚入口

你不应该再被迫理解：

- 哪个窗口是 `Team Pool`
- 哪个窗口是 `V2`
- 哪个旧脚本该不该点
- 现在到底是 health 问题还是 synthetic 问题

### 6.2 Team Pool 应该退到后台

目标不是删掉 Team Pool，而是把它变成：

- `V2` 管理的后台执行引擎
- 不再需要你手工开窗口盯着看
- 不再让你把它当主界面使用

### 6.3 旧入口层改成可选组件

像这些：

- `anthropic_proxy.js`
- `cloudflared.exe`
- `start_tunnel.bat`
- `start_anthropic_proxy.bat`

不应该继续跟“本机自用模式”绑死。  
应该只在“对外服务模式”里启用。

---

## 7. 平台模式设计

建议把平台运行画像拆成 2 套：

### 7.1 本机自用模式 `local_single_user`

目标：

- 只保留你自己真的会用到的能力

只需要：

- `Team Pool :8317`
- `V2 Gateway :18320`
- `/ops`
- `Health Probe`
- `Synthetic Probe`

默认不要求：

- Cloudflare Tunnel
- Anthropic Proxy `8320`
- New API `3001`
- 外部入口链路

适合场景：

- 你自己本机调用
- 你自己调试和观察
- 你不需要给别人或外部设备用

### 7.2 对外服务模式 `public_service`

目标：

- 保留完整入口能力

包括：

- Team Pool
- V2
- Anthropic Proxy
- Tunnel
- 外部入口 / 客户端兼容层

适合场景：

- 你要给别人用
- 你要给外部设备用
- 你要保留公网入口

### 7.3 这一步的价值

一旦加上运行画像，你的 `/ops` 页面就不会再总是把“对外服务模式”的要求强塞给“本机自用模式”。

这是提升体验的关键一步。

## 8. 单平台版 V2 的页面蓝图

### 8.1 首页只展示 5 个最重要的状态

首页顶部只保留：

1. 当前模式
2. 当前是否 ready
3. 可路由账号数
4. 底层引擎状态
5. 最近体检状态

### 8.2 页面主结构建议

#### A. 总览区

- 当前平台模式：本机自用 / 对外服务
- 当前切流状态：legacy / parallel / canary / primary
- 当前就绪状态：ready / blocked
- 最近建议：继续运行 / 先重跑体检 / 先回滚

#### B. 运行控制区

把现在散落在脚本里的操作收进来：

- 启动本机环境
- 停止本机环境
- 重启底层引擎
- 重启 V2
- 一键体检
- 回滚到 legacy

#### C. 体检区

只保留最有用的 3 个动作：

- 运行账号同步
- 运行健康巡检
- 运行链路体检

并把结果用一句人话说明：

- 哪项没通过
- 为什么没通过
- 下一步应该点什么

#### D. 底层引擎状态区

重点展示：

- Team Pool 是否存活
- Team Pool 是否能成功完成 completion
- 上一次失败原因
- 上一次恢复动作

#### E. 历史与审计区

保留但折叠：

- recent synthetic probes
- recent events
- recent operator actions

这样新手不用一上来就被很多表格淹没。

## 9. 脚本收编蓝图

### 9.1 现在应该收编的脚本

优先收编：

- `start_team.bat`
- `start_v2.bat`
- `enter_parallel.ps1`
- `enter_canary.ps1`
- `enter_primary.ps1`
- `rollback_legacy.ps1`
- `health_check.ps1` 里的“只与 V2 / Team Pool 相关的部分”

### 9.2 收编方式

不是删除，而是：

- 继续保留脚本作为底层 helper
- 但在 `/ops` 里提供统一按钮调用它们

这样可以做到：

- 平台统一操作
- 底层仍然复用已验证过的脚本

### 9.3 不建议继续暴露给你的入口

以后不建议再让你直接面对：

- `start_all.bat`
- `start_anthropic_proxy.bat`
- `start_tunnel.bat`

这些应该退到：

- 对外服务模式专用
- 或平台内部封装调用

## 10. 进程管理蓝图

### 10.1 平台应该直接看见这两个核心进程

必须直接管理：

- `Team Pool :8317`
- `V2 Gateway :18320`

### 10.2 应该新增的能力

建议在控制层里加：

- 检查端口是否监听
- 检查进程是否存在
- 启动进程
- 停止进程
- 重启进程
- 记录最近一次重启原因

### 10.3 真正解决你痛点的点

你现在觉得烦，不是因为页面不够美，而是因为：

- 进程管理在窗口里
- 状态判断在页面里
- 规则在脚本里

把这三者收回 V2，才是真正的平台合并。

## 11. 推荐的三阶段落地路线

## 阶段 A：单平台外壳

### 目标

让你以后只用 `/ops`。

### 交付物

- `/ops` 新增“平台模式”切换
- `/ops` 新增“启动本机环境 / 停止本机环境 / 重启 Team Pool / 重启 V2”
- `/ops` 新增“底层引擎状态”
- `/ops` 把旧入口层状态默认折叠

### 这一步不做的事

- 不重写 Team Pool
- 不删 `cli-proxy-api.exe`
- 不改认证引擎

### 验收标准

- 你自己不再需要手工双击 `start_team.bat` 和 `start_v2.bat`
- 你自己不再需要盯多个窗口来判断系统是否正常

---

## 阶段 B：本机自用模式收口

### 目标

让系统真正适配“只自己用”的场景。

### 交付物

- `local_single_user` 模式
- 本机模式下：
  - 不检查 tunnel
  - 不检查 anthropic proxy
  - 不要求对外链路
- readiness 规则切换为：
  - Team Pool completion 可用
  - V2 synthetic 可用
  - 账号可路由

### 验收标准

- 你本机自用时，页面不再被一堆与外网无关的状态干扰
- 你只要看 `8317 + 18320 + readiness`

---

## 阶段 C：底层引擎替代评估

### 目标

评估 Team Pool 是否可被 V2 真正吞并。

### 先要回答的几个问题

1. Team Pool 当前的认证态到底从哪里拿
2. `auth_unavailable: no auth available` 的真正根因是什么
3. 哪些执行逻辑只存在于 `cli-proxy-api.exe`
4. 是否能把这些能力外移成可控模块

### 这一步的结论可能有两种

#### 方案 C1：继续保留 Team Pool 当后台引擎

如果 Team Pool 是稳定黑盒，就让它留在后台，V2 负责平台层。

#### 方案 C2：开始逐步吞并执行能力

如果 Team Pool 的认证与执行逻辑能被拆出来，再考虑：

- 账号级 active routing
- 去掉 `cli-proxy-api.exe`
- 真正变成 V2 单引擎平台

### 注意

这是未来阶段，不是你当前最优先的事。

## 13. 我对你这个项目的最终建议

### 不建议现在做的

- 不建议一上来“彻底干掉 V1”
- 不建议现在就试图把 `cli-proxy-api.exe` 完整重写
- 不建议继续维持“V1 一套入口，V2 一套入口，你自己脑子里切换”

### 强烈建议现在做的

1. 让 V2 变成唯一可见平台
2. 让 Team Pool 退到后台执行
3. 增加本机自用模式
4. 把脚本收编成平台内部能力

### 你以后应该达到的体验

- 只打开一个页面
- 只看一个状态面板
- 只点一个启动入口
- 出问题时只点一个回滚按钮

这才是你要的“大平台”。
