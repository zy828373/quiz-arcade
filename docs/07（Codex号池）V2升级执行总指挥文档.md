# 07（Codex号池）V2升级执行总指挥文档

> 文档定位：本文件是“总指挥文档”，用于你在 VSCode 的 Codex 插件中指挥执行代理分阶段落地号池 V2 升级。

> 文档目标：把当前号池从“脚本堆叠式单机系统”升级为“统一网关 + 账号状态机 + 带权调度 + 指标账本 + 运维知识化”的可维护系统。

## 一、角色分工

### 1. 总指挥

你在当前聊天中扮演总指挥，职责是：

- 决定当前要推进哪个阶段
- 决定是否接受某个架构取舍
- 审核执行代理的结果与风险说明
- 在跨阶段或高风险变更前做最终拍板

### 2. 执行代理

VSCode 中打开本仓库的 Codex 插件扮演执行代理，职责是：

- 先读文档和现有代码，再动手
- 每次只完成一个阶段或一个明确工单
- 实施代码修改、脚本调整、文档补充和验证
- 汇报改了什么、怎么验证、还有什么风险

### 3. 协作原则

- 总指挥只下发一个清晰阶段，不要一次让执行代理横跨多个阶段
- 执行代理默认先保守重构，不做无授权的大拆大改
- 所有高风险动作必须先说明影响，再执行
- 所有输出必须围绕“可上线、可回滚、可验证”

## 三、当前系统的工程判断

当前仓库已经具备可用链路，但核心问题不在“能不能转发”，而在“能不能稳定演进”。

### 1. 当前链路

- `cli-proxy-api.exe`：主号池能力
- `anthropic_proxy.js`：Anthropic 协议转换
- `cloudflared.exe`：公网入口
- `health_check.ps1`：保活与粗粒度恢复
- bat 脚本：循环拉起服务
- `auths_team/`、`auths_free/`：账号文件存储

### 2. 当前主要短板

- 入口层职责重叠，协议适配、路由、恢复逻辑分散
- 路由只有轮询，没有账号健康分级和调度策略
- 没有正式的账号状态机
- 没有持久化指标账本，只能从日志事后猜
- 恢复逻辑偏“杀进程重启”，缺少幂等恢复
- 运维知识散落在脚本和交接文档里，难以长期维护

### 3. 升级核心原则

- 不先推翻现有链路，先旁路建设 V2，再逐步接管
- 不先重写 `cli-proxy-api.exe`，先补齐控制面和调度面
- 不把复杂 agent 编排塞进生产请求路径
- 先做“可观测、可判断、可恢复”，再做“更智能”

## 四、V2 目标架构

V2 的目标不是“换一份脚本”，而是形成四层结构。

### 1. 流量面 Traffic Plane

职责：

- 鉴权
- 协议适配
- 请求标准化
- 将请求交给调度器

要求：

- 一个主网关为主入口
- OpenAI / Anthropic / 其他协议通过 adapter 方式接入
- 不允许多个入口同时持有独立路由逻辑

### 2. 池控制面 Pool Control Plane

职责：

- 账号注册表
- 账号状态机
- 带权调度器
- 熔断与冷却
- 健康探测
- 审计与指标账本

### 3. 运维面 Ops Plane

职责：

- 结构化日志
- 实时状态视图
- 告警与恢复动作记录
- 手工隔离与恢复
- 故障复盘数据

### 4. 知识面 Knowledge Plane

职责：

- 维护 Runbook
- 沉淀故障模式
- 把重复诊断动作模板化
- 给未来代理和维护者提供标准入口

## 五、技术路线约束

为了降低改造风险，执行代理按下面约束落地。

### 1. 技术选型

- 新增 V2 逻辑优先使用 Node.js + TypeScript
- 状态存储优先使用 SQLite
- 日志优先使用结构化 JSON
- 配置优先使用 `.env` + 明确配置文件

选择理由：

- 当前已有 Node.js 自研组件，迁移阻力最低
- SQLite 足够支撑单机控制面，运维成本低
- 先单机稳定，再谈多机或容器化扩展

### 2. 边界约束

- 第一阶段不替换 `cli-proxy-api.exe`
- 第一阶段不依赖修改外部 Docker 中的 New API
- 第一阶段不碰 Cloudflare 生产域名策略
- 第一阶段不移动现有敏感账号文件目录

### 3. 代码组织建议

建议执行代理在仓库内新增独立目录，例如：

```text
v2/
  src/
    adapters/
    config/
    control/
    gateway/
    health/
    ledger/
    logging/
    pool/
    routing/
  data/
  scripts/
  tests/
```

说明：

- V2 先作为并行系统开发
- 旧脚本继续保底可用
- 新系统成熟后，再逐步替换旧脚本入口

## 六、账号状态机设计基线

执行代理在实现调度器前，必须先把账号状态定义清楚。

建议状态：

- `active`：正常可分配
- `warming`：新账号或恢复中账号，仅小流量试探
- `cooldown`：短期失败过多，临时冷却
- `degraded`：可用但表现差，降权
- `quarantined`：疑似风控或异常，隔离观察
- `expired`：明确过期
- `disabled`：人工禁用
- `archived`：退出生产池，仅归档

建议状态迁移触发：

- 认证失败、权限失败、限流、网络错误、超时、成功恢复
- 人工操作，如手动隔离、手动恢复、手动归档

要求：

- 每次状态变化必须留事件记录
- 每个状态变化都必须能说明原因
- 调度器只能基于状态和指标做决策，不能写死文件名特判

---

## 七、调度器设计基线

V2 的调度器必须从 `round-robin` 升级为“健康加权调度”。

最少要具备：

- 成功率权重
- 延迟权重
- 最近错误惩罚
- 冷却与熔断
- 恢复试探流量
- 能力标签过滤

建议决策顺序：

1. 先按协议和模型能力筛账号
2. 再过滤不可用状态
3. 再按健康分数排序
4. 对恢复中的账号只给小流量
5. 失败后立即记录惩罚并视情况切换状态

调度器必须输出可解释信息：

- 为什么选这个账号
- 为什么没选其他账号
- 当前权重和惩罚项是什么

## 八、升级分阶段路线

执行代理必须严格按阶段推进，前一阶段没验收，不进入下一阶段。

### 阶段 0：建立 V2 基线骨架

目标：

- 在不影响现网的情况下建立 V2 工程目录和运行骨架

执行内容：

- 新建 V2 目录结构
- 初始化 TypeScript 工程
- 建立统一配置读取模块
- 建立统一日志模块
- 建立 SQLite 初始化脚本
- 建立基础 HTTP 服务骨架
- 补一份 V2 README

交付物：

- 可启动的 V2 空骨架服务
- 清晰的目录结构
- 基础配置样例
- 基础日志输出

验收标准：

- 不影响现有 `start_all.bat` 链路
- V2 服务可以本地启动
- 启动后有 `/health` 或类似健康接口
- 有最基础的单元测试或启动验证

### 阶段 1：账号注册表与状态账本

目标：

- 把散落的账号文件信息同步进正式控制面

执行内容：

- 扫描 `auths_team/`、`auths_free/`
- 设计 SQLite 表结构
- 建立账号元数据表
- 建立账号状态事件表
- 建立同步脚本
- 为账号生成稳定 ID，不依赖文件名临时解析

交付物：

- 账号注册表
- 同步脚本
- 初始导入结果
- 状态事件模型

验收标准：

- 执行同步后能列出当前账号清单
- 每个账号有明确状态
- 账号变更会进入事件表
- 不修改原始 token 文件结构

### 阶段 2：健康检查引擎与指标采集

目标：

- 从“进程是否活着”升级成“账号和链路是否真的健康”

执行内容：

- 设计账号级健康检查
- 设计服务级健康检查
- 设计隧道级健康检查
- 将结果写入指标表和事件表
- 给现有 `health_check.ps1` 提供更细粒度的判定依据

交付物：

- 健康检查模块
- 指标采集表
- 健康状态汇总接口

验收标准：

- 能区分账号异常、网关异常、上游异常、隧道异常
- 不再只用“是否能访问端口”做唯一健康判断
- 健康检查结果可追踪

### 阶段 3：带权调度器与熔断机制

目标：

- 用健康加权路由替代单纯轮询

执行内容：

- 实现路由评分模型
- 实现状态过滤
- 实现熔断和冷却
- 实现恢复试探流量
- 给出调度决策日志

交付物：

- 新调度器
- 调度结果解释
- 路由测试样例

验收标准：

- 连续失败账号会自动降权或隔离
- 恢复账号不会立刻吃满流量
- 调度器可在 shadow mode 运行并记录结果

### 阶段 4：统一网关与协议适配层

目标：

- 把协议适配逻辑从单文件脚本升级为可扩展 adapter 架构

执行内容：

- 抽离当前 `anthropic_proxy.js` 的协议转换逻辑
- 定义标准请求模型
- 定义标准响应模型
- 适配 OpenAI 风格入口
- 适配 Anthropic 风格入口
- 统一鉴权和超时策略

交付物：

- 新主网关
- 至少两个 adapter
- 兼容现有客户端的接口说明

验收标准：

- 能覆盖当前 `anthropic_proxy.js` 主要能力
- 新旧网关可并行测试
- 不要求第一时间替换旧入口，但必须可验证

### 阶段 5：控制面 API 与运维面板

目标：

- 给维护工作建立可视化入口和可操作入口

执行内容：

- 提供账号列表接口
- 提供状态变更接口
- 提供事件查询接口
- 提供健康汇总接口
- 提供最小可用 Web 面板或静态页面

交付物：

- 控制面 API
- 最小运维面板

验收标准：

- 可查看账号状态、最近错误、健康摘要

- 所有人工操作有审计记录

### 阶段 6：旧脚本瘦身与链路切换

目标：

- 让旧脚本从“核心逻辑承担者”退化成“启动壳”

## 十三、参考这次升级所借鉴的工程思想

本次 V2 方案不是凭空想象，而是综合了三类工程思路：

- `pi-mono`：学习其分层清晰、扩展点明确、adapter/provider 解耦的方式
- `gsd-2`：学习其状态机、指标账本、恢复幂等、控制面优先的方式
- `oh-my-claudecode`：学习其 hook、runbook、知识沉淀、任务模板化的方式

注意：

- 抄的是工程原则，不是整套产品表面形态
- 对当前号池最重要的是“控制面”和“可维护性”，不是做花哨编排

---

## 十四、最终交付目标

当 V2 升级完成时，系统应该达到以下状态：

- 有一个明确主入口
- 有一套正式账号注册表
- 有一套账号状态机
- 有一个带权调度器
- 有一套指标和事件账本
- 有一个最小控制面
- 有成体系的维护文档和执行模板

如果执行代理的工作偏离以上目标，总指挥应立即拉回，不要让它在无关重构上消耗时间。

---

## 十五、一句话总纲

这次升级的核心不是“把现有脚本改漂亮”，而是把号池从“脚本拼装系统”升级成“有控制面、有状态、有调度、有审计、能持续维护”的工程系统。
